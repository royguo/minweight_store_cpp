# AI Agent 文档目录指南

**重要提示**：所有 AI Agent 在本项目中开始任何任务之前，必须阅读并遵守本文档。

`agents_read.md` 所在目录就是项目文档根目录。本文档中的所有路径，除明确说明外，都相对于该文档根目录。

人类不会手动维护文档根目录下的项目文档。保持文档结构清晰、状态准确、引用有效，是 AI Agent 的责任。

## 目录结构与使用规则

文档根目录是项目架构、设计、任务状态和操作记录的唯一真实来源。当前目录职责如下：

### 1. `design/`：稳定架构与设计

- **用途**：统一存放已经确认或正在作为实现依据的系统架构、模块设计、技术方案。
- **入口**：优先阅读 `design/index.md` 和相关模块设计文档。
- **规则**：
  - 不再使用独立的 `architecture/` 目录；高层架构属于 `design/`。
  - 新模块、大规模重构、状态机、恢复流程、并发一致性路径，必须在实现前或实现同步更新对应设计文档。
  - 未定案的方案讨论不要放在 `design/`，应先放入 `discuss/`。

### 2. `discuss/`：设计讨论

- **用途**：存放复杂设计讨论、取舍分析、可行性评估、评审材料和还没有定案的技术路线。
- **入口**：优先阅读 `discuss/index.md`。
- **规则**：
  - 讨论文档应说明当前状态：`Open`、`Converted`、`Superseded` 或 `Resolved`。
  - 讨论完成后，如果形成可执行工作，必须在 `trackers/todos.md` 中创建或更新任务，并在任务中引用对应 discuss 文档。
  - 如果讨论已经沉淀为稳定设计，应把最终结论同步到 `design/`，同时在 discuss 文档中标注对应设计文档和任务状态。
  - `discuss/` 不是无差别历史归档目录；无持续参考价值的历史文档应合并到 `trackers/resolved.md` 或当天 worklog。

### 3. `trackers/`：任务列表

- **用途**：只维护任务状态列表。
- **文件**：
  - `todos.md`：所有当前未完成任务，包括 bug、功能、重构、测试、文档债务和可靠性缺口。
  - `resolved.md`：所有已完成 TODO 和已解决 bug 的归档。
- **规则**：
  - `trackers/` 下不再新增单任务详档。
  - 发现新问题或新技术债时，加入 `todos.md`。
  - 完成或解决任务后，从 `todos.md` 移到 `resolved.md`，保留完成日期、解决方案和验证信息。
  - 当被要求“寻找下一个任务”或“检查当前优先级”时，始终检查 `trackers/todos.md`。

### 4. `worklogs/`：操作日志

- **用途**：只记录实际操作过程，不承载设计方案或长期任务状态。
- **命名规范**：按日期命名，文件名必须是 `YYYY-MM-DD.md`，标题必须是 `# YYYY-MM-DD`。
- **规则**：
  - 每一轮 commit 之前，必须把本轮操作内容追加到当天 `worklogs/YYYY-MM-DD.md`。
  - 日志记录应包括操作内容、关键文件、验证命令/结果、tracker/design 更新情况。
  - 不再使用 `WIP-主题` 或 `日期-主题` 文件名。
  - 详细设计和讨论不得写入 `worklogs/`；应分别写入 `design/` 或 `discuss/`。

### 5. `guides/`：开发者指南

- **用途**：人类可读的项目介绍、贡献规则和性能调优手册。
- **规则**：除非任务明确涉及指南维护，否则很少更新。

### 6. `skills/`：Agent 辅助规则

- **用途**：补充性的 Agent / coding 规则。
- **规则**：如果与本文档冲突，以 `agents_read.md` 为准；若形成新的长期约定，需要同步更新本文档。

## 文档写作规则

1. **文档语言**：文档内容统一使用中文编写。后续所有新增文档和对既有文档的新增段落都必须使用中文；代码、API、命令、路径、协议字段、错误码、日志关键字、状态枚举等专有名词可保留英文。修改既有英文段落时，应优先改写为中文，除非该段落是外部引用、命令输出或协议原文。
2. **示例说明**：编写架构说明、技术实现说明、问题说明、故障复盘和任务说明时，应尽量举例说明。示例可以是最小调用流程、状态迁移、输入输出、失败日志、配置片段或复现步骤，帮助读者把抽象描述映射到具体场景。
3. **注释语言**：所有注释使用英文，包括代码注释、脚本注释、配置注释、Markdown 注释和文档中的示例代码注释。
4. **引用路径**：文档中引用本文档根目录内的文件时，使用相对文档根目录的路径，例如 `design/index.md`、`trackers/todos.md`、`worklogs/YYYY-MM-DD.md`。
5. **表格输出**：面向用户或文档的表格应优先使用适合命令行终端阅读的等宽文本表格，例如 fenced `text` 代码块中的 ASCII 表格；需要注意列宽一致、内容对齐和行宽可读，避免默认 Markdown 表格在终端中过宽或错位。

## 代码规则

1. **测试文件位置**：新测试优先遵循项目既有测试布局；没有更明确约定时，与被测试源代码文件放在同一级目录。不要为单个需求临时新建独立顶层 `test/` 目录。历史遗留测试目录在没有明确指令时保持不变。
2. **代码风格规范**：代码必须符合当前项目的语言、框架和既有风格。C++ 项目默认遵循 Google C++ Style Guide。单元测试名称遵循项目既有命名；没有约定时使用大驼峰命名法，不使用下划线作为分隔符。
3. **注释语言**：新增或重构注释必须使用英文；修改已有非英文注释时，优先同步改为英文。
4. **关键代码注释**：新写或重构后的重要函数、复杂状态机、恢复流程、并发/一致性路径，必须在函数或关键代码段附近说明实现步骤、状态语义和失败处理方式。
5. **架构文档同步**：如果实现中发现或引入架构调整，必须更新 `design/` 下对应设计文档。
6. **配置参数传递**：代码设计中应尽量避免在业务模块、manager、helper 和纯函数中直接读取 flags。可配置参数应优先在 config loader、bootstrap 或上层 subsystem options 中解析，再通过 `Options`、构造参数或函数参数显式向下传递，避免 flags 泛滥和隐式全局状态。
7. **提交前文档检查**：每一轮 commit 前，必须检查并更新相关 `design/`、`discuss/`、`trackers/todos.md`、`trackers/resolved.md` 和当天 `worklogs/YYYY-MM-DD.md`。
8. **临时测试目录**：临时测试或运行产物必须放在项目内 `.runtime/` 或项目约定的运行目录，除非用户明确要求或工具无法配置，不要写入 `/tmp/<project>` 等项目外目录。
9. **问题复现优先级**：遇到问题应优先使用 UT 构造最小复现；如果状态机、多进程时序或外部依赖导致 UT 复杂度过高或难以可靠模拟，可以改用确定性的 integration test 复现。
10. **Chaos 失败复现路径**：chaos test 暴露的新问题，应先按“问题复现优先级”补充确定性复现，再推进产品修复；不要通过放宽 chaos oracle 掩盖问题。

## Agent 标准操作程序

1. **上下文初始化**：先定位并读取文档根目录中的 `agents_read.md`。
2. **状态同步**：检查 `trackers/todos.md`，必要时查看 `design/index.md`、`discuss/index.md` 和相关设计/讨论文档。
3. **执行与记录**：
   - 不要求用户管理文档。
   - 随任务进展自主更新合适的文档文件。
   - 复杂任务的讨论过程写入 `discuss/`，稳定设计结论写入 `design/`，操作过程写入 `worklogs/YYYY-MM-DD.md`。
4. **任务完成**：
   - 如果完成了 tracker 项，把它从 `todos.md` 移到 `resolved.md`。
   - 确保设计决策、验证结果和当天操作日志已经落盘。
